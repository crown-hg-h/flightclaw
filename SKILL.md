## 低价出行方案思想（预规划）

做「便宜且可行」的出行方案，核心是在 **总成本**、**总时间**、**衔接风险** 之间权衡，而不是只看一张机票票面价。

### 可比性原则

- **同一维度对比**：比较不同方案时，尽量统一口径，例如「机票展示价 + 机建燃油等常见附加」「高铁二等座参考价」「必要的市内接驳时间」。
- **时间也要计价**：中转或空铁联运若节省数百元但多出半天与两次误机风险，是否值得由用户偏好决定；Agent 应 **并列给出时间与费用**，而非只报最低价。
- **余票与价格非承诺**：工具拉取的是检索页展示信息，**以各平台实时下单为准**。

### 三类典型路径

| 类型 | 含义 | 典型场景 |
|------|------|----------|
| **直飞** | 出发机场 → 目的地机场，一段航班 | 枢纽对枢纽、航线多、比价简单 |
| **航空中转** | 经第三城换机，一张订单或分段多张票 | 枢纽少、直飞贵；或刻意选低价中转组合 |
| **高铁 + 机票（空铁联运）** | 先铁路到 **有机场的枢纽城市**，再飞目的地 | 小城市直飞贵/班次少；或铁路段短、航班段选择多 |

### 中途下车策略（经停/中转站下车）

**含义**：你真正要去的是 **A → B**，但 **A → B 直飞（或同一张票里「名义终点为 B」）很贵**；而检索 **A → C** 的航班时，若 **经停站或中转站恰好是 B**，且 **整张联程/拼接票价比 A→B 更便宜**，则可以在 **B 结束行程、不再乘坐后续航段**——你把它称为「中途下车」。

**典型例子**：丽江 → 北京 报价高；同一日期检索 **丽江 → 沈阳**（**在北京中转或经停**），总票价反而更低——你的实际目的地仍是北京，在 **北京** 下机后放弃北京→沈阳段。

**比价时怎么做**：在 flightclaw 中除 `A→B` 外，**额外扫** 若干「更远但经停 B」的目的地 C（需看列表是否展示经停/中转城市）；把 **展示总价 + 时间与衔接** 与直飞/常规中转对比。

**必须向用户说明的风险与限制**（规划文档里应写明，避免当作「常规低价技巧」）：

- **运输合同与航司规则**：多数航司条款不鼓励「仅使用部分联程」；违规使用可能影响后续服务，**以各航司公布条件为准**。
- **行李**：若行李 **托运至最终目的地 C**，通常 **无法在中途站 B 正常提取**；仅随身、或分段自行处理行李时才现实。
- **往返与联程**：同一订单内若后段弃乘，可能影响 **返程或同一票号下的后续航段**；航班 **签转、取消、保护航班** 时，航司可能改到 **不经停 B** 的路线，计划会失效。
- **仍须满足输出规范**：若推荐此类思路，须 **分项标价、时间轴、衔接与弃程说明**，并强调 **以支付页与航司规则为准**。

### 总成本与总时间（心算框架）

- **总成本（示意）** ≈ 航班相关支出 + 铁路相关支出 + 合理估算的接驳/税费差异（国内线常有机建燃油，分段购票可能重复计）。
- **总时间** ≈ 铁路运行时间 + 站到机场接驳 + 值机安检缓冲 + 飞行及中转等待 + 到达后路程。  
  中转或空铁联运时，**衔接缓冲**（尤其跨站、跨航站楼）应保守估计。

### 预售期与「参考日期」（铁路尤其重要）

- **每次检索前先取「当天」日期**（与机票一致：可用 `flightclaw today` 或本机 `Asia/Shanghai` 的 `YYYY-MM-DD`），再与用户 **目的乘车日** 对比。
- **12306 高铁票**常见规则是 **仅预售未来约 15 天内**（具体以 12306 公布为准）。若 **目的乘车日距当天超过约 15 天**，该日 **尚未开售**，用脚本按该日查询往往 **无结果、异常页或风控**，**不要**把失败当成「没有车次」。
- **远期规划时的做法**：铁路段改用 **「当天日期 + 10 天」** 作为 `query.mjs -d` 的 **参考乘车日**。高铁 **G/D 等固定运行图线路**在 **周内平日** 的 **班次时刻、停站与耗时** 通常高度重复，**参考日** 用于给出 **班次密度、大致时刻区间与二等座价位量级**；须在输出中标明 **「参考日 ≠ 用户实际出行日」**，临近预售开放后再用 12306 核对 **真实日期** 的余票与票价。
- **若目的乘车日已在预售期内**（距当天 ≤ 约 15 天）：应 **直接** 用该目的日 `-d 用户日期` 查询；仍失败则按 12306 排障（风控、站名）处理。
- 机票侧日历与铁路预售 **可能不一致**，须 **分别说明**。

## Agent 最终输出规范（必须满足）

用户要的是 **可执行的方案对比**，而不是零散报价。在给出推荐或方案列表时，**每一条方案** 都应同时包含下面两类信息（可用表格 + 时间轴文字）。

### 1. 时间规划（必须具体）

- **日期**：出行日（若铁路与航班日期不同须分别写清）。
- **分段时刻**：铁路写出 **车次或大致发/到时刻区间**；航班写出 **计划起飞、计划到达**（中转写清每段与 **中转停留**）。
- **接驳与缓冲**：高铁站 → 机场、跨航站楼、跨站换乘，至少写明 **建议预留分钟数**（如「站→机场 ≥90min」），并说明是否同城异站。

### 2. 价格规划（必须具体）

- **分项标价**：高铁（二等座参考或展示价）、每段机票（展示起价/票面）、若可估算则单列 **机建燃油** 或注明「以支付页为准」。
- **合计**：该方案 **总价区间或加总**（说明是否含税、是否含铁路）。
- **口径说明**：标注数据来自 **携程列表 / 12306 查询 / 参考日期**，避免与「最终支付价」混淆。

### 3. 多方案汇总表（推荐格式）

当同时给出 **多类可比对方案**（直飞 / 航空中转 / 空铁 / 相邻日出行的改期备选 / 多枢纽出发等）时，在已满足上文 **时间 + 价格** 分项说明的前提下，**宜用一张 Markdown 总表** 收敛对比，避免关键数字散落在多段叙述中。

| 要素 | 要求 |
|------|------|
| **列建议** | **方案**（类型简述）；**路线**（航班号、大致航段或「城市高铁 + 机场代码→机场代码」）；**日期**；**计划时刻与总耗时**（铁路写车次或区间亦可）；**展示价**（分项或合计，注明是否含铁路）；**说明**（衔接缓冲、数据来源、参考日≠实际日等） |
| **推荐标注** | 对 **主推荐方案** 在表内使用 **粗体**（可 **整行加粗**，或仅对「方案」「展示价」等关键格加粗）。若存在 **互斥约束**（例如「必须指定日期」与「可改期」各有一条最优），可 **各加粗一行**，表后附 **一两句摘要** 说明取舍 |
| **表前口径** | 表上方用简短一句话说明 **价格/时刻来自**携程 DOM、VariFlight、12306 等**哪一来源**，并重申 **以各平台支付页为准** |

**示例结构**（字段名可按实际增删；加粗行为演示）：

```markdown
**口径**：航班展示价来自〈数据源〉；机建燃油与实付价以支付页为准。铁路若用参考日，见说明列。

| 方案 | 路线 | 日期 | 计划时刻 / 总耗时 | 展示价（约） | 说明 |
|------|------|------|-------------------|--------------|------|
| **直飞（推荐·总价）** | **WUX→LJG，XX0000** | **YYYY-MM-DD** | **09:30→14:40，约 5h** | **约 N 元** | **当日同源低价；少中转** |
| 直飞（省时） | WUX→LJG，YY0000 | YYYY-MM-DD | 09:35→12:45，约 3h | 约 M 元 | 耗时最短，价高 |
| 空铁 | 无锡→南京（高铁）+ NKG→LJG | … | … | 机票 + 高铁 | 备选：直飞售罄/涨价时 |
```

若某段无法查到（如铁路超出预售期），仍须在输出中 **显式写出缺口** 与建议用户何时再用 12306 复核。

## 携程查询前：必须提示 Cookie / 会话路径

使用 **`--provider ctrip`**（或任何依赖携程 Playwright 的检索）时，常遇风控、空白列表或超时。执行查询前，Agent **应主动提示用户** 配置本机 Cookie 或登录态文件（**勿将密钥提交 Git**）：

| 环境变量 | 含义 |
|----------|------|
| **`FLIGHTCLAW_CTRIP_COOKIES_FILE`** | **优先提示用户给出**：Chrome 扩展等导出的 Cookie **JSON 文件路径**（本机绝对路径）。导出后仅本地使用。 |
| **`FLIGHTCLAW_CTRIP_STORAGE_STATE`** | Playwright `storage_state` JSON 路径（与上二选一即可，若已用此项可不设 Cookie 文件）。 |

**话术示例**：「请提供本机携程 Cookie JSON 的路径，并在终端执行：`export FLIGHTCLAW_CTRIP_COOKIES_FILE='/你的路径/ctrip-cookies.json'` 后再运行检索。」

未提供 Cookie 仍可直接查询，但若结果异常，应 **再次提示** 补充路径后重试。

| **空铁联运**（先高铁到枢纽再飞） | **两者配合**：先用 12306 算铁路段时间与票价量级，再用 flightclaw 查 **枢纽机场** 出发的航班 | 枢纽需与用户确认（如上海虹桥/浦东、北京首都/大兴等），注意 **车站 ≠ 机场** 的接驳 |

| **中途下车**（真要去 B，但扫 A→更远 C 经停 B 是否更便宜） | 除 `A→B` 外对若干 **C** 做 `search`；核对列表中的 **经停/中转** 是否为 B | 须按上文「中途下车策略」写明 **条款、行李、往返与签转风险** |
| **方法论、话术、评估表**（多模态旅行、空铁案例） | 可选阅读 [`skills/openclaw-import/`](skills/openclaw-import/) 下相关摘录 | **不执行查询**，仅供规划思路参考 |

**不推荐混淆**：不要用 12306 技能查机票；不要用 flightclaw 查火车票。铁路与民航数据源与命令完全不同。

### Agent 检索清单（可选）

同一机票比价需求下，Agent 可按下表 **主动多扫几眼**，减少「用户未提才漏掉更优枢纽」的情况（如 **只查了无锡、被追问才补南京**）。

| 场景 | 建议 |
|------|------|
| **长三角 → 滇西**（丽江 LJG、大理、版纳等） | **至少** 用 `search` 扫 **WUX** 与 **NKG**；若涉及空铁去上海飞，**必要时** 再扫 **SHA→LJG**（或 **PVG→LJG**，与虹桥/浦东衔接分开说明）。 |
| **节假日 / 长假前后** | 对用户给定出行日，**相邻 1～2 天** 各查一次航班（如 **5.1 vs 5.2**），日历价差与余票波动大时尤应如此。 |
| **枢纽** | 用户只说了小城市出发时，仍可按上文 **「枢纽机场与周边城市」** 小节表格，**主动纳入** 常见比价枢纽（如 **上海、昆明**），并在输出里说明是否值得多一段铁路/接驳。 |

**目的**：在 **用户未逐一点名**「南京 / 上海」时，也能给出 **可对比的备选出发面**，避免遗漏。

### 12306 脚本（经验摘要）

- **日期**：先取当天，再判断用户乘车日是否 **超出约 15 天预售**；超出时 **`query.mjs -d` 用「当天+10 天」** 作参考日（见上文「预售期与参考日期」），输出注明参考口径。
- 余票接口为 **两步**（先 `query` 再按 JSON 内 **`c_url`** 请求）；Node **`fetch` 若默认跟随重定向** 可能拿到 HTML，出现 **`Unexpected token '<'`**——以 **[`skills/12306-skill/SKILL.md`](skills/12306-skill/SKILL.md)** 中「实战经验与排障」为准。
- 若第二步被重定向到 **`error.html` / 空响应**，多为 **风控**：应提示用户用 **12306 官网或 App**，并在方案说明中标明「铁路段未拉取」。
- 一城多站务必先用 **`stations.mjs`** 再 **`query.mjs`** 精确站名；空铁联运注意 **高铁站与机场** 是否同城、接驳时间。

## 何时使用本仓库（汇总）

- 需要 **脚本化 / JSON** 拉取 **航班** 或 **列车** 信息，或按上表做 **空铁组合比价**。
- 用户表达 **便宜、中转、经大城市飞、高铁到机场、中途下车/经停下车** 等，可按「三类路径 + 中途下车策略 + 技能表」拆解任务。

## 何时不用

- **下单、占座、支付** → 走官方 App/网站或授权渠道；本仓库工具 **不完成交易**。
- 仅需 **随手打开网页** → 直接使用携程、12306 等即可。

## 安装（摘要）

```bash
cd skills/flightclaw-skill && pip install -e ".[dev,ctrip]" && playwright install chromium
```

入口：`flightclaw` 或 `python -m flightclaw`；未安装时可在仓库根执行 **`python skills/flightclaw-skill/scripts/flight.py`**。默认无子命令进入 **REPL**；全局 **`--json`** 输出结构化结果。

**机票（仓库根，免安装示例）**：

```bash
python skills/flightclaw-skill/scripts/flight.py --json search --from BJS --to SHA --date 2026-04-01 --provider example
```

**铁路（需 Node.js）**：

```bash
node skills/12306-skill/scripts/query.mjs 北京 上海 -t G -f md
node skills/12306-skill/scripts/stations.mjs 杭州
```

详见 [`skills/12306-skill/SKILL.md`](skills/12306-skill/SKILL.md)。

## 输出约定（机票 JSON）

- 成功：JSON 含 `provider`、`query`、`offers[]`（`flight_no`、`dep_time`、`arr_time`、`price`、`currency`、`note`、`raw`）、`warnings[]`。
- **携程等为页面展示起价**，以官网为准。
